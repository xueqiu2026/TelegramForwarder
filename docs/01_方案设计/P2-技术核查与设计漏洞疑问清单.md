## 一、 核心业务逻辑漏洞（导致数据重叠/失效）

### 1.1 定时总结消息范围重叠 Bug
*   **出处代码**：[scheduler/summary_scheduler.py:L142-L154](file:///D:/Axiangmu/TelegramForwarder/scheduler/summary_scheduler.py#L142-L154)
*   **具体问题**：
    在现有的消息获取逻辑中，定时总结的开始时间 `start_time` 被硬编码为前推 **24 小时**：
    ```python
    start_time = now.replace(
        hour=summary_hour,
        minute=summary_minute,
        second=0,
        microsecond=0
    ) - timedelta(days=1)
    ```
    一旦根据 P2 设计方案在 `summary_time` 中配置了多时间点（如每天 4 次：`"06:00,12:00,18:00,00:00"`），如果获取时间范围依然前推 24 小时，**相邻两次总结的内容将产生 18 小时的重叠**（例如 12:00 总结会包含 06:00 已总结过的消息）。这不仅会导致用户重复阅读，更会导致重复向 AI 接口发送相同的历史消息，造成巨额 Token 浪费。
*   **推荐改进方案**：
    在支持多时间点后，`start_time` 必须改为**“本次执行时间点”与“上一总结时间点”的差值**（即拉取最近 6 小时内的新消息），而不是每次都拉取过去 24 小时的数据。

## 二、 系统崩溃与代码阻塞隐患（导致进程无法启动/卡死）

### 2.1 多时间点 `summary_time` 解析崩溃
*   **出处代码**：[scheduler/summary_scheduler.py:L116-L124](file:///D:/Axiangmu/TelegramForwarder/scheduler/summary_scheduler.py#L116-L124)
*   **具体问题**：
    调度器计算下一次运行时间的逻辑是：
    ```python
    def _get_next_run_time(self, now, target_time):
        hour, minute = map(int, target_time.split(':'))
        ...
    ```
    一旦在 `summary_time` 字段中存入逗号分隔的多时间点字符串（如 `"06:00,12:00,18:00,00:00"`），`split(':')` 将产生 5 个以上的切片，导致 `map(int, ...)` 抛出 `ValueError: too many values to unpack`，**直接导致整个定时总结调度器在冷启动时崩溃，无法工作**。
*   **推荐改进方案**：
    需要将解析逻辑解耦，改为识别逗号分割的列表，并支持遍历列表中所有时间点以计算出临近的最小值。

### 2.2 多定时任务管理导致的“幽灵任务”残留
*   **出处代码**：[scheduler/summary_scheduler.py:L24](file:///D:/Axiangmu/TelegramForwarder/scheduler/summary_scheduler.py#L24) 和 [L38-L42](file:///D:/Axiangmu/TelegramForwarder/scheduler/summary_scheduler.py#L38-L42)
*   **具体问题**：
    现有系统中的 `tasks` 变量采用 `{rule_id: task}` 一对一存储协程任务：
    ```python
    if rule.id in self.tasks:
        old_task = self.tasks[rule.id]
        old_task.cancel()
    ```
    如果按照 P2 方案为每条规则的每个时间点单独注册一个 asyncio 任务，`tasks` 的字典结构将被破坏或覆盖。当用户关闭某规则的总结功能或修改配置时，**系统将只能取消其中一个任务，而其余时间点的总结任务将成为“幽灵任务”常驻后台**，继续抓取数据并调用 AI 接口发送。
*   **推荐改进方案**：
    仍然维持 `{rule_id: task}` 的一对一简洁管理关系。在单个 `_run_summary_task(rule)` 的 `while` 循环内部，动态计算距离当前时间最近的下一个时间点（Next Run Time），计算出等待秒数进行 `sleep`。每次触发总结后计算下一个时间点，无需为每个时间点单独生成协程。

### 2.3 PostgreSQL 迁移中的 SQLite 特有 DDL 迁移兼容性
*   **出处代码**：[models/models.py:L405-L409](file:///D:/Axiangmu/TelegramForwarder/models/models.py#L405-L409) 以及 [L414-L459](file:///D:/Axiangmu/TelegramForwarder/models/models.py#L414-L459)
*   **具体问题**：
    现有的 `migrate_db` 逻辑包含大量为 SQLite 规避缺少 ALTER 约束支持而编写的代码，如查询系统表 `sqlite_master` 检查约束是否存在，以及通过“创建临时表 `keywords_temp` -> 导数据 -> DROP 物理表 -> 重命名”的 SQLite 特有变通逻辑。
    **如果直接连接 PostgreSQL 执行这段代码，进程将在冷启动迁移时抛出 SQL 语法错误并直接异常退出**。
*   **推荐改进方案**：
    增加数据库类型识别逻辑（例如检查连接串是否包含 `postgresql`）。若为 Postgres，则直接通过 ANSI SQL 标准的 `ALTER TABLE` 命令直接修改或增加唯一索引约束，跳过 SQLite 特有的临时表机制。

## 三、 系统架构与部署优化疑问

### 3.1 前端控制台与现存 RSS 后端服务的端口合并
*   **出处代码**：[main.py:L111-L129](file:///D:/Axiangmu/TelegramForwarder/main.py#L111-L129)
*   **具体问题**：
    目前 RSS 服务 `rss/main.py` 在独立进程中由 Uvicorn 启动，占用 8000 端口（Docker 内部映射为 9804）。若为了控制台再独立部署一个 API 服务，将面临多进程管理、多端口占用及前端跨域配置繁琐的问题。
*   **推荐改进方案**：
    将前端控制台规划的 24 个 API 路由与鉴权逻辑直接并入现有的 `rss/` 服务中（可整体更名为后端 `backend` 服务），统一使用 8000 端口，前端通过同一端口通信，避免跨域和端口冲突。

### 3.2 热重载心跳协程的数据库轮询开销
*   **出处配置**：P2 设计方案 7.3 节
*   **具体问题**：
    方案提到“在守护进程中增加心跳，每 60 秒比对数据库变更”以触发热重载。在多频道、高频转发的场景下，60 秒轮询略显滞后，且无谓地增加了数据库连接和查询压力。
*   **推荐改进方案**：
    由于数据库已经迁移到 PostgreSQL，可利用 Postgres 的 `LISTEN/NOTIFY` 机制。当 Web 服务端完成配置修改并写入库后，触发 `NOTIFY`，主监听进程通过 `LISTEN` 立即感知并触发规则重载（`scheduler.schedule_rule()`），实现零开销、零延迟的实时热重载。

## 四、 消息推送兼容性隐患

### 4.1 国内平台（如企业微信）的消息字数与格式限制
*   **出处代码**：[scheduler/summary_scheduler.py:L241-L248](file:///D:/Axiangmu/TelegramForwarder/scheduler/summary_scheduler.py#L241-L248)
*   **具体问题**：
    AI 总结生成的 Markdown 内容通常较长，且带有特殊标记。国内平台（如企业微信群机器人）的消息接口存在严格的字数限制（普通文本限制 2048 字节，Markdown 限制 4096 字节）。直接向国内平台推送未经清洗的 TG 格式 Markdown 消息，**极易触发接口字数超长报错而被平台拒收，或者由于格式不兼容导致乱码**。
*   **推荐改进方案**：
    在推送到企业微信/飞书/钉钉前，应增加格式适配器（Adapter），自动精简 Markdown 语法，并在超出字数上限时进行截断或拆分发送。
