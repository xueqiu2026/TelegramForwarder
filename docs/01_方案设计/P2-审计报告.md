| 方案描述 | 实际代码证据 |

**修正建议**：将描述改为"现有 TG Bot 回调路径已具备热重载能力，但新增的 Web 前端直写数据库路径需要补建独立的变更感知机制"。

## 三、风险项（建议关注）

### 3.1 SQLite 并发写入锁

P2 方案设计了"TG Bot + Web 前端双重写入同一 SQLite 数据库"的架构。SQLite 在写入时会锁定**整个数据库文件**（Write-Ahead Logging 模式下可缓解但不能完全消除）。当守护进程的 Telethon 客户端与 FastAPI Web 服务同时执行写操作时，存在 `sqlite3.OperationalError: database is locked` 的风险。

**建议**：
- 启用 WAL（Write-Ahead Logging）模式：`PRAGMA journal_mode=WAL`
- 在所有 `create_engine` 处统一设置 `connect_args={"timeout": 30}`

### 3.2 第七章 PostgreSQL 兼容性：遗漏 `ALTER TABLE RENAME COLUMN`

方案仅提到了双引号默认值问题，但 [models.py:L410](file:///D:/Axiangmu/TelegramForwarder/models/models.py) 中的迁移脚本还使用了 `ALTER TABLE forward_rules RENAME COLUMN mode TO forward_mode`。此语法在 PostgreSQL 12 以下版本不支持（需要使用 `ALTER TABLE ... RENAME ... TO ...` 的不同格式），但 PostgreSQL 12+ 已支持，应在方案中明确最低 PG 版本要求。

### 3.3 硬编码 DB 路径：存在 2 处而非 1 处

方案第七章仅示例了 `init_db()` 中的修改，但实际 `models.py` 中有 **2 处**独立的硬编码：
- [L464](file:///D:/Axiangmu/TelegramForwarder/models/models.py#L464)：`init_db()` 函数
- [L476](file:///D:/Axiangmu/TelegramForwarder/models/models.py#L476)：`get_session()` 函数

两处都需要统一改为 `os.getenv('DATABASE_URL')` 驱动。
